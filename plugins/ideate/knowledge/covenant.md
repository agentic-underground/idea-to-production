# IDEATE — pillars & the KAIZEN self-improvement covenant

> This plugin's own anchor for the marketplace's governing philosophy. Like every `idea-to-production`
> plugin, IDEATE is bound by the **three pillars** and the **KAIZEN self-improvement covenant**. (The
> canonical, in-depth homes live in the `foundry` plugin's `knowledge/`; this is the local copy a
> standalone install carries — referenced by concept, not by a cross-plugin path.)

## The three pillars

- **Knowledge-parity** (≡ *knowledge-alignment*) — understand the ask completely **before** acting; a
  recurring question becomes a written answer, asked once. This is IDEATE's *whole purpose*: refine an
  idea until a fresh agent with no history can act on it without guessing.
- **Quality-first** (≡ *quality-confidence*) — quality is built in, not inspected in; a gate is never
  weakened to make progress. IDEATE's gate is the **IDEA-package exit criteria** — do not hand off a
  package that doesn't satisfy them, however eager the user is to start.
- **Waste-elimination** (≡ *muda · mura · muri*) — remove waste in every form, *including rediscovery*. Every ambiguity resolved
  is written into the package so it is never asked again; a fuzzy brief handed downstream is the most
  expensive waste there is, because every station amplifies it.

> **Overarching constraint — token-efficiency:** thin skills, fat references; define once, reference
> many; the agent-facing package is *simplified and high-precision* precisely so the conveyor's context
> stays tight.

## The KAIZEN self-improvement covenant

Every element of this plugin continuously asks how it can improve, and each iteration must at least
**halve the remaining distance to perfection**; an element that grows to do more than one thing
**self-cleaves** into smaller, single-purpose elements.

Concretely, IDEATE improves by folding **downstream feedback** back into its challenge protocol and
package contract: when a builder hits an ambiguity the IDEA package *should* have resolved (a vague
actor, an untestable success metric, an unstated constraint), that becomes a sharper challenge axis or a
new package field — landed via branch → adversarial review → **PR**, so **every future ideation, for all
users, asks the missing question by default**. The `self-improve` skill drives this loop.

> A package that satisfied its exit gate yet still produced a downstream ambiguity has not honoured the
> covenant — and the fix is upstream: a better-articulated challenge axis, fixed once.
