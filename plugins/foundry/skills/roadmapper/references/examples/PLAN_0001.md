# Add a task (title + complete toggle)

**Brief**: The first vertical slice of TaskFlow — a user types a title and adds a task; it appears in
the list and persists across a reload, carrying a `done` flag that starts `false`. This slice also
creates the shared **Task store** the rest of EPIC_0001 reuses.

## User stories
- AS A person with things to remember I WANT to add a task by typing its title SO THAT it is captured
  in one trustworthy place.
- AS A returning user I WANT a task I added earlier to still be there SO THAT I can trust the list not
  to lose my work.

## EARS specification
- **EARS-001** (Event-driven) — WHEN the user submits a non-empty task title, the system SHALL create a
  task with that title, a unique id, and `done = false`, and append it to the task list.
- **EARS-002** (Unwanted) — IF the user submits an empty or whitespace-only title, THEN the system
  SHALL reject the submission and SHALL NOT create a task.
- **EARS-003** (Ubiquitous) — The system SHALL persist every created task to the task store so that it
  survives process restart and page reload.
- **EARS-004** (Event-driven) — WHEN the task list is loaded, the system SHALL display every persisted
  task with its title and completion state.

## Acceptance criteria
1. Given an empty list, When the user adds a task titled "Buy milk", Then a task "Buy milk" appears in
   the list with `done = false`.
2. Given the user submits an empty title, When they attempt to add, Then no task is created and the
   list is unchanged.
3. Given a task "Buy milk" was added, When the application is reloaded, Then "Buy milk" is still
   present in the list with its `done` state preserved.
4. Given two tasks were added, When the list is read back, Then both appear, each with a distinct id.

## Human interface test plan
- **Add-task field + button.** Navigate to the TaskFlow main view → find the "New task" text input and
  the "Add" button → verify both visible → type "Buy milk" → click "Add" → verify a list row reading
  "Buy milk" with an unchecked completion control appears → reload the page → verify the "Buy milk" row
  is still present and still unchecked. (This gesture is consumed by the EARS/FEATURE/STORY phases so a
  browser-level story test pins it.)

## Implementation notes
- Components likely affected: a new `Task` model (`id`, `title`, `done`); a new `TaskStore` repository
  (JSON-file-backed `add` / `list`) — this is the EPIC's shared persistence layer (EPIC_0001 §Shared
  infrastructure), so build it as the reusable seam, not a one-off.
- Patterns to reuse: the project's existing id-generation utility if present; otherwise a UUID.
- Known risks: concurrent writes to the JSON store (out of scope for this single-user slice — note it,
  don't solve it here); title trimming/normalisation (trim before the empty-check of EARS-002).
- External dependencies / API contracts: none beyond the local task store.
