**Brief**: Multiple backends (a pool of ComfyUI workers) behind one MCP endpoint, and a cache keyed by
`(template_id, params, checkpoint)` so identical figures are free.

## User stories
- AS an operator under load I WANT several ComfyUI workers behind one MCP endpoint SO THAT generation
  scales, AND AS a frequent caller I WANT identical requests cached SO THAT repeat figures are instant and
  free.

## EARS specification
- `EARS-001` (Ubiquitous): THE SYSTEM SHALL distribute generation across a pool of ComfyUI workers behind
  one MCP endpoint and cache results by `(template_id, params, checkpoint)`.
- `EARS-002` (Event-driven): WHEN a request matches a cached `(template_id, params, checkpoint)` key THE
  SYSTEM SHALL return the cached result without re-generating.

## Acceptance criteria
1. Given multiple workers, When several requests arrive, Then they are distributed across the pool.
2. Given an identical prior request, When repeated, Then the cached result is returned (no re-generation).

## Human interface test plan
None — pooling/caching are server internals. Proof: concurrent requests are distributed across stubbed
workers; a repeat of an identical `(template_id, params, checkpoint)` returns the cached result with no
re-generation (asserted on the dispatch + cache, stubbed workers — deterministic).

## Implementation notes
- Pool behind the one MCP endpoint; cache keyed by `(template_id, params, checkpoint)`. Builds on the
  Phase-1 server (PLAN_0046). Cache-key equality must be exact so a differing param never serves a stale
  figure.
- Stubbed workers in tests (no live ComfyUI) — deterministic, no flakiness.
