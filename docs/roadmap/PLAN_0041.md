**Brief**: A pan/zoom canvas of connected objects (nodes + edges) on a very large surface — with spatial
keyboard navigation, an accessible non-spatial alternative, level-of-detail (LOD) rendering, and INTENT
markers that survive on canvas nodes. Architecturally event-driven + space-based.

## User stories
- AS a user exploring a large graph I WANT to pan/zoom and navigate by keyboard SO THAT the canvas is
  usable without a mouse, AND I WANT an accessible non-spatial alternative SO THAT the view isn't
  spatial-only.

## EARS specification
- `EARS-001` (Ubiquitous): THE SYSTEM SHALL render a pan/zoom node-edge canvas with level-of-detail and
  keyboard spatial navigation.
- `EARS-002` (Event-driven): WHEN the canvas zooms out THE SYSTEM SHALL reduce node detail (LOD) while
  keeping nodes selectable; WHEN a node is focused by keyboard THE SYSTEM SHALL scroll it into view.
- `EARS-003` (Optional): WHERE a purely spatial view is insufficient THE SYSTEM SHALL offer an accessible
  structured (list/tree) alternative over the same graph.

## Acceptance criteria
1. Given a large graph, When zoomed/panned, Then LOD keeps it responsive and nodes remain selectable.
2. Given keyboard-only use, When navigating, Then nodes are reachable and an accessible non-spatial view is
   available.

## Human interface test plan
- **[canvas pan/zoom + keyboard]** load a large node-edge graph → zoom out → assert LOD reduces detail yet
  nodes stay selectable → keyboard-focus a node → it scrolls into view → toggle the structured
  (list/tree) alternative → the same graph is navigable non-spatially (browser-level story test).

## Implementation notes
- Event-driven + space-based architecture — see the design system's architecture-styles guidance under
  `plugins/foundry/skills/frontend/`.
- INTENT markers must survive on canvas nodes; honour the accessibility/keyboard rules.
- No arbitrary timing waits in tests — poll for the LOD/focus state (the FLAKY-TEST ban).
