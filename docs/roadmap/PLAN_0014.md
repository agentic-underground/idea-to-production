# Define the MCP exposure policy + opportunity scan

**Brief**: Define and enforce which MCP actions agents may call, and add a capability to scan a project
for MCP opportunities and present them to the owner as steerable choices. Cheap/common verbs (read/query,
stage transitions, telemetry) are exposed by default; expensive authoring/graph mutations are gated behind
explicit approval.

## User stories
- AS the owner I WANT cheap/common MCP actions exposed and expensive ones gated SO THAT agents are fast
  for routine flow while I keep control of costly mutations.
- AS the owner I WANT a scan that surfaces candidate MCP actions as choices SO THAT I can steer where the
  MCP surface is headed.

## EARS specification
- `EARS-001` (Ubiquitous): THE SYSTEM SHALL expose roadmap-read/query, stage-transition, and telemetry
  verbs to agents by default.
- `EARS-002` (Optional): WHERE a verb is classed *expensive* (e.g. an authoring `request_rewrite` or a
  graph `mutate_connection`) THE SYSTEM SHALL require explicit approval before invocation.
- `EARS-003` (Event-driven): WHEN the owner runs the MCP-opportunity scan THE SYSTEM SHALL present
  candidate actions, grouped common vs. expensive, as a choice list for approval/steering.
- `EARS-004` (Unwanted): IF an agent calls a gated (expensive) verb without approval THEN THE SYSTEM
  SHALL refuse it with a clear reason.

## Acceptance criteria
1. Given the exposure policy, When an agent calls a gated verb without approval, Then it is refused with
   a clear, human-readable reason.
2. Given the scan, When run, Then it lists candidate MCP actions grouped by common vs. expensive for the
   owner to choose from.
3. The policy is documented as a single source of truth and reflected in settings/registration.

## Human interface test plan
- **[opportunity scan]** run the scan → an AskUserQuestion-style choice list appears, partitioned into
  *common* and *expensive* candidates → the owner selects which to expose/gate → the selection is
  persisted to the policy source.
- **[gated refusal]** with the policy active, an agent invokes an expensive verb without approval → the
  call is refused → the refusal states which verb and why.
