**Brief**: PR #56 shipped three behaviours into the FOUNDRY lifecycle *outside* the SDLC — no EARS, no
Gherkin, no story proof. This slice runs those behaviours through the full quality chain retroactively so
each is pinned by a test coordinate and regressions are caught at test time, not in production.

## User stories
- AS the FOUNDRY system I WANT every shipped lifecycle-delivery behaviour pinned by EARS + a test
  coordinate SO THAT regressions are caught at test time, not discovered in production.

## EARS specification
- `EARS-001` (Event-driven): WHEN the lifecycle history loads a record whose status is `AWAITING MERGE`
  THE SYSTEM SHALL map it to a terminal `Done` status on startup.
- `EARS-002` (Event-driven): WHEN `ds-step-9-commit-push` completes its commit/push action THE SYSTEM
  SHALL post a `done` status for the item to the canvas surface (Action #8).
- `EARS-003` (State-driven): WHILE a PLAN's PR is open and unmerged THE SYSTEM SHALL hold the
  lifecycle-orchestrator in an `AWAITING MERGE` pause rather than reporting the item complete.
- `EARS-004` (Event-driven): WHEN the PR for a paused item is merged THE SYSTEM SHALL run the post-merge
  completion handler that transitions the item to complete.
- `EARS-005` (Unwanted): IF the canvas/status surface is unreachable WHEN Action #8 posts THE SYSTEM
  SHALL degrade gracefully (the commit/push must not fail because the status sink is down).
- `EARS-006` (Unwanted): IF the post-merge handler runs against a PR that is not yet merged THEN THE
  SYSTEM SHALL NOT mark the item complete.

## Acceptance criteria
1. Given the three PR #56 behaviours, When the retrofit is complete, Then an EARS id exists for each and
   every EARS id maps to at least one Gherkin scenario.
2. Given the `AWAITING MERGE → Done` startup mapping, When the unit suite runs, Then the mapping is
   covered by an explicit test, expanded to its unhappy/abuse paths (unknown status, malformed record).
3. Given the `AWAITING MERGE` pause + post-merge handler, When exercised, Then there is at least one
   happy (PR merged → complete), one unhappy (merge fails → stays paused), and one abuse (not-yet-merged
   PR → refuses to complete) scenario, each backed by a story-level proof.
4. Given Action #8 (status sync), When exercised, Then a scenario exists for both the sink-up and
   sink-down paths.
5. Coverage floor for the touched code is 100% (every behaviour pinned; the floor that results from
   pinning, not the target).

- **No new production behaviour is intended** — this is spec + Gherkin + test authoring that pins
  behaviour PR #56 already shipped.
- **CRITICAL provenance caveat — re-scope before building.** Behaviours `EARS-001` and `EARS-002` were
  authored against the **`flow` plugin** (`history.rs` in `flow-server`; `post_status` on the flow
  canvas), which has since been **RETIRED** (commit `da07db8` — "retire the flow plugin; the FLEET engine
  supersedes it"). Before writing tests, the builder MUST re-locate these behaviours on the **current**
  surface: the `AWAITING MERGE → Done`/`completed` mapping now lives in the FLEET engine's manifest
  `state` column semantics, and the status post is whatever the current lifecycle emits. Pin the
  behaviour that exists today; do not resurrect retired `flow-server` code. Behaviour `EARS-003`/`004`
  (the lifecycle-orchestrator pause + post-merge handler) still lives in
  `plugins/foundry/agents/lifecycle-orchestrator` and is directly testable.
- **Test homes:** Gherkin `.feature` scenarios alongside the existing lifecycle features; agent-
  instruction story tests invoke `lifecycle-orchestrator` against a fixture roadmap item and assert the
  sentinel chain + emitted status calls.
