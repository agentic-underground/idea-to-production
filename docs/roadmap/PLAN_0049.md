**Brief**: The `container/` build — ComfyUI + the MCP server, bind-mounting `DIFFUSION_MODELS`, putting
ComfyUI on a private network, and publishing **only** the MCP port. Realises the network-isolation half of
PLAN_0048.

## User stories
- AS an operator I WANT a container that runs ComfyUI + the MCP server with the model drive bind-mounted
  and only the MCP port exposed SO THAT the backend is reproducible and the raw ComfyUI API is never
  reachable.

## EARS specification
- `EARS-001` (Ubiquitous): THE SYSTEM SHALL build a container running ComfyUI + the MCP server, with
  `DIFFUSION_MODELS` bind-mounted and ComfyUI on a private network.
- `EARS-002` (Event-driven): WHEN the container is brought up THE SYSTEM SHALL publish only the MCP port
  and keep ComfyUI unpublished.
- `EARS-003` (Unwanted): IF the compose/run config would publish the ComfyUI port THEN THE SYSTEM SHALL be
  considered misconfigured (only the MCP port may be published).

## Acceptance criteria
1. Given the container, When up, Then ComfyUI sees the bind-mounted models and only the MCP port is
   published.
2. Given a port scan of the host, When run, Then the ComfyUI API port is not reachable.

## Human interface test plan
None — the container is infra. Proof: a config assertion that only the MCP port is published (ComfyUI
unpublished, on the private network) and the model drive is bind-mounted; a host port-scan test confirms
the ComfyUI API port is unreachable.

## Implementation notes
- `comfyui-mcp/container/` builds both services; private network; bind-mount the model drive. Realises the
  network-isolation half of PLAN_0048. The compose/run config publishing the ComfyUI port is the
  misconfiguration the abuse-path test catches.
- If a Dockerfile/compose is added, keep it consistent with the marketplace's IaC hygiene (the security
  EPIC_0029 container/IaC lens ethos): no root where avoidable, pinned base image.
