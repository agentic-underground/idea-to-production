**Brief**: Browse grids, tables, and spreadsheet-dense paradigms need windowing/virtualization to stay
fast and accessible at thousands of rows — preserving keyboard roving, focus management, and reduced
layout shift. Affects `browse-grid`, a future `data-table`, and `inline-text-edit` at scale.

## User stories
- AS a user of a data-dense grid I WANT thousands of rows to scroll smoothly SO THAT the UI stays fast,
  AND I WANT keyboard navigation + focus to keep working SO THAT virtualization doesn't break
  accessibility.

## EARS specification
- `EARS-001` (Ubiquitous): THE SYSTEM SHALL render large row sets via windowing/virtualization while
  preserving keyboard roving and focus management.
- `EARS-002` (Event-driven): WHEN a virtualized row scrolls out of and back into view THE SYSTEM SHALL
  restore its focus/selection state and avoid layout shift.
- `EARS-003` (Unwanted): IF virtualization would drop a focused row from the DOM THEN THE SYSTEM SHALL
  keep focus reachable (retain/re-bind it) rather than lose the user's place.

## Acceptance criteria
1. Given thousands of rows, When scrolled, Then frame rate stays smooth and layout shift is minimal.
2. Given keyboard navigation, When rows virtualize in/out, Then roving focus and selection are preserved.

## Human interface test plan
- **[virtualized scroll + keyboard]** load a grid with thousands of rows → scroll → assert windowed
  rendering and minimal layout shift → roving-keyboard down through the set → a row that virtualizes out
  and back retains focus/selection (a browser-level story test via the design system's test harness).

## Implementation notes
- Applies to `browse-grid`, a future `data-table`, and `inline-text-edit` (the shared windowing discipline
  the registry-growth elements in PLAN_0044 consume).
- Preserve the design system's keyboard/focus/INTENT conventions under windowing — see
  `plugins/foundry/skills/frontend/`.
- No arbitrary timing waits in tests — poll for the specific DOM/focus state (the FLAKY-TEST ban).
