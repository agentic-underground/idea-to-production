# Add diff-scoped mode (/security:scan-all --since <ref>)

## Metadata
| | |
| --- | --- |
| **Plan** | `0036` |
| **Epic** | `0029` |
| **Depends on** | — |

**Brief**: Scan only what changed since a git ref (`--since main`) for fast, incremental gating on every
PR — clearly labelled as partial coverage so it is never mistaken for a full audit.

## User stories
- AS a developer I WANT the gate to scan only my diff SO THAT PR feedback is fast instead of re-scanning
  the whole tree every time.

## EARS specification
- `EARS-001` (Ubiquitous): THE SYSTEM SHALL support scanning only the files changed since a given git ref.
- `EARS-002` (Event-driven): WHEN `/security:scan-all --since <ref>` runs THE SYSTEM SHALL restrict each
  lens to the changed files and report findings only within that diff.
- `EARS-003` (Unwanted): IF a finding's risk extends beyond the diff (e.g. a secret added in an earlier
  commit on the branch) THEN THE SYSTEM SHALL disclose that diff-scoped mode may miss it, so it is not
  mistaken for a full scan.

## Acceptance criteria
1. Given a branch with changes, When `--since main` runs, Then only changed files are scanned and the run
   is faster than a full scan.
2. Given diff-scoped mode, When it completes, Then the output states it was diff-scoped (not a full audit).

## Human interface test plan
None — `--since <ref>` is a CLI flag. Proof: against a fixture branch, only the changed-file set is scanned
and the output is labelled PARTIAL/diff-scoped (asserted on the scanned-file list + the report banner).

## Implementation notes
- Compute the changed-file set via `git diff --name-only <ref>`; pass it to each lens.
- **Always label diff-scoped results as PARTIAL coverage** (the EARS-003 disclosure) so they are not read
  as a full audit.
- **Rename caveat:** the gate is `/security:scan-all` (legacy `/security-gate`).
