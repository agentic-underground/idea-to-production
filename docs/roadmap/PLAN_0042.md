**Brief**: The open problem from the design system's privacy-as-architecture philosophy: let a user move
between phone and laptop **without a server**, keeping data private and local-first, without the friction
of manual export/import every time. Explore CRDTs, peer-to-peer transports, and encrypted-blob handoff —
cloud-grade continuity with local-first privacy.

## User stories
- AS a privacy-conscious user I WANT to continue my work when I switch from phone to laptop SO THAT I get
  cloud-grade continuity without my data passing through a server.

## EARS specification
- `EARS-001` (Ubiquitous): THE SYSTEM SHALL support transferring a user's working state between two of
  their devices without a central server, keeping the data local-first and private.
- `EARS-002` (Event-driven): WHEN a user moves to another device THE SYSTEM SHALL hand off the working
  state (via a peer transport or encrypted-blob exchange) without a manual per-move export/import.
- `EARS-003` (Unwanted): IF a transport is unavailable THEN THE SYSTEM SHALL fall back to an explicit
  encrypted handoff rather than route data through a server.

## Acceptance criteria
1. Given two devices of one user, When work continues on the second, Then the state transfers without a
   server and without manual export/import.
2. Given the handoff, When inspected, Then data is encrypted in transit and stored local-first.

## Human interface test plan
- **[device handoff]** simulate two device contexts → continue work on the second → assert the working
  state transferred with no server hop and no manual export/import → inspect the payload → it is encrypted
  in transit and persisted local-first (story test against simulated peers; no real network).
