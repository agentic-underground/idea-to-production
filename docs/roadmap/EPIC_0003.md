- **The authoring value task** (`plugins/foundry/skills/roadmapper/SKILL.md` Â§3.5) is the engine each
  PLAN runs: roadmapper orchestrates `builder-lead` + `handler-roadmap-decomposition` to carve a batch

- **The conformance gate** (`scripts/verify-prereqs.sh` check P) + the vendored grammar

  PLAN's output â€” the new docs must parse under the engine grammar.
- **Global id allocation** is shared: each PLAN claims the next free `NNNN` across all `EPIC_`/`PLAN_`
  files (this EPIC is `0003`; its PLANs `0004`â€“`0009`; new migrated items continue from `0010`). PLANs
  run in order to avoid id collisions on the shared counter.
- **Legacy provenance:** when a batch is migrated, its source `.i2p/roadmap/backlog/*` files stay in
  place as history (per the migration decision) â€” the v2 docs are the live surface; the tree is the
  archive.

## Definition of done
- Every legacy `.i2p/roadmap/backlog/` item is represented by a conformant v2 EPIC/PLAN doc (or folded
  into an existing one), recorded in each PLAN's mapping.
- `scripts/verify-prereqs.sh` check P is GREEN over the full `docs/roadmap/` tree; `.pipeline/verify`
  passes.
- The `.pipeline.md` manifest lists every emitted EPIC; no legacy item is silently dropped.

## Execution contract (for the builder)
Each PLAN decomposes ONE theme's legacy items via the authoring value task and emits the v2 docs; a
failed PLAN leaves its batch un-migrated (the legacy files remain authoritative for that batch) and is
resumed next round. PLANs are ordered so the shared id counter never collides.
