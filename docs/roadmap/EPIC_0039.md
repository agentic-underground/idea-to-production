## Shared infrastructure
- **The `foundry:frontend` design-system surface** — the element registry (registry-page anatomy +
  `elements/README.md` discoverability via `-help`), the INTENT-marker convention, and the
  accessibility / keyboard-roving / dark-mode rules every element must honour. Every PLAN below extends
  this one surface; none re-establishes it. The virtualization work (PLAN_0040) underpins the future
  `data-table` element in the registry-growth umbrella (PLAN_0044), so its windowing/focus discipline is
  the shared piece those elements consume.
- **The adversarial design-critic** — PLAN_0043 automates spawning the critic (the `atelier`
  design-reviewer lens) and routing its findings into a scored loop; the registry-growth elements
  (PLAN_0044) are graded by that same critic's definition-of-good.

## Definition of done
- All five PLANs are `completed` (the engine rolls EPIC completion up from its PLANs).
- `.pipeline/verify` is GREEN for each landed PLAN; `scripts/verify-prereqs.sh` (incl. check P, and check
  J four-mirror for any new skill/registry surface) stays GREEN.

## Execution contract (for the builder)
The engine builds one PLAN at a time in `## Plans` order, running each PLAN's `## Construction process`
(FOUNDRY scoped to the `(EPIC, PLAN)` pair). A failed PLAN leaves its row `engaged` and resumes next tick;
the EPIC lands only when all five PLANs are GREEN. PLAN_0042 (serverless device-swap) is **research-grade**
— it may itself spawn sub-slices once its transport approach is chosen (see that PLAN).
