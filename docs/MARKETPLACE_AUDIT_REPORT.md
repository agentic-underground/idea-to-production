# idea-to-production Marketplace — Capability Audit Report

## Executive Summary

The **idea-to-production** marketplace is, at the documentation-and-contract level, a remarkably complete and internally-coherent software production house. It spans nine lifecycle phases — DISCOVER ▸ IDEATE ▸ DELIVER ▸ DESIGN ▸ BUILD ⇄ ASSURE ⇄ SECURE ▸ PUBLISH ▸ OPERATE ↻ — across eight composable, self-contained plugins (the **i2p** front door plus seven specialists). Its single strongest, genuinely-working capability is the **BUILD engine**: FOUNDRY ingests a roadmap, tiers items by token budget, and drives each through a disciplined 0–9 SDLC conveyor via a cleanly-layered orchestration hierarchy (founder → builder-lead → lifecycle-orchestrator → value-handlers → reviewer), where every agent declares its altitude and the "implement, don't orchestrate" boundary is enforced uniformly.

The audit's headline finding is a sharp gap between **prose and runtime wiring**. The marketplace is strongest precisely where its contracts are explicit and weakest precisely where the live wiring has not yet caught up to the narrative. The flagship loop that the README, VALUE_FLOW, and product-lifecycle docs all advertise — **BUILD ⇄ ASSURE ⇄ SECURE** — is **not traversable end-to-end at runtime**: two of the nine documented lifecycle transitions have no caller anywhere in the marketplace. After FOUNDRY advances `done BUILD` → ASSURE, nothing advances ASSURE → SECURE, and the `fail` back-edge — the entire mechanism by which a failed quality or security gate re-enters BUILD — is never invoked by any plugin. The loop is real on paper and dead in the wiring.

Beyond the central loop defect, the audit surfaced a cluster of **catalog-vs-disk drift** (commands advertised that have no command file, a retired `flow` plugin still named as a live owner, a delegation to a non-existent `/publish:design-review`), several **value-chain seams** where producer and consumer schemas don't match (the OPERATE ↻ DISCOVER re-entry artifact, the standalone IDEA-package filesystem schema), and **handler-coverage** gaps (an orphaned on-disk Ansible handler, no documented stack-detection algorithm, large swathes of mainstream stacks with no handler). None of these are fatal to the architecture — the design is sound — but together they break the journey for a user holding a raw value-proposition who tries to travel idea → production today.

The value-handlers are a high-quality, template-consistent fleet (test-first mandate, 100% coverage floor, per-phase model-tiering, ADR discipline) covering the Python/JS/TS/React/Rust families well.

Three pieces of machinery stand out as exemplary. The missing-handler gate replaces silent degradation with an explicit 3-way decision (build-handler / MVP-degraded / both). The graceful-degradation discipline is woven through every seam — each cross-plugin call is guarded "by capability", absent companions degrade to annotated markdown rather than failing silently. And the BUILD-internal handoff layer (typed context-sentinel chain + strict YAML handoff schema + per-phase checkpoints) genuinely lets a cold-start agent resume mid-pipeline.

The DELIVER→FLEET handoff is the single most fully-closed contract: a version-pinned EPIC/PLAN/.pipeline grammar with a registry schema and a GREEN-verify floor. PUBLISH's illustrator demonstrates a model dispatch table (intent→handler with named tie-breaks) the foundry stack-pool should emulate. The lifecycle state machine (lifecycle.sh) is sound and order-safe by design. And the self-inspection lenses provably catch real drift (publish's inspector already flagged a naming error). The marketplace is strongest where contracts are explicit and weakest precisely where the runtime wiring has not yet caught up to the prose.

## Genuine Strengths

- **VALUE_FLOW.md is a genuinely strong operational spine** — an 11-station table (IDEA→VERIFY) with explicit input-contract / owning-skill / staffed-by / exit-gate columns, a layered orchestration hierarchy (founder→builder-lead→lifecycle-orchestrator→ds-step/handler→reviewer), and a master index (§10) cross-referencing skills↔agents↔knowledge. The ds-step-0..9 agent files all exist on disk as claimed.
- **Skill frontmatter is uniformly high-quality across all 8 plugins** — every SKILL.md carries a name, a rich trigger-laden description, and a metadata block (type/output/composes). The `composes:` fields make cross-plugin orchestration auditable (scan-all composes pii/secrets/dependencies; pr-review composes reviewer+scan-all; operate-gate composes observability/incident/iterate/maintain).
- **The three-verb convention (check/inspect/self-improve) is genuinely consistent** — all 8 plugins ship check + inspect + self-improve, and SLASH_COMMANDS.md correctly factors them out as "Common to every plugin" rather than relisting.
- **The DELIVER→FLEET handoff contract is exceptionally well-specified and version-pinned** (roadmapper/references/fleet-pipeline-standard.md, 241 lines): EPIC→PLAN→change decomposition, the `~/.claude/pipeline-projects.json` registry schema, the GREEN `.pipeline/verify` floor, and an explicit named Construction-process bridge handing each PLAN to FOUNDRY's PLAN-scope entry. The most fully-closed contract in the chain.
- **The missing-handler-gate is a genuinely well-designed governance protocol** — it replaced silent detect-and-degrade with an explicit 3-way decision gate (BUILD HANDLER FIRST / MVP+DEGRADED_CAPABILITIES / BOTH), wires the awaiting-handler↔DEFERRED pairing through roadmapper idioms, and pushes the cheapest catch upstream to IDEATE's stack-fit challenge.
- **Graceful degradation is consistently and explicitly handled at every plugin boundary** — each `done <PHASE>` call is guarded "by capability", the DEGRADED_CAPABILITIES contract gives a real detect→route→disclose→never-false-pass discipline, and absent companions degrade to markdown with a stated note rather than silent failure.
- **The lifecycle state model itself (lifecycle.sh done/fail/advance/set) is well-designed** — order-safe and idempotent, so a misfiring plugin can never jump the lifecycle out of order. The state machine is SOUND — the defect is purely that two transitions have no caller, not that the model is broken.
- **Every foundry handler is internally consistent and disciplined** — identical Test-First Mandate, 100% line+branch coverage floor, a Spawning Model Policy table (haiku/sonnet/opus per phase), the "tests are coordinates" canon, and a closing handler-annotation + KAIZEN covenant.
- **The orchestration hierarchy is cleanly layered and each agent states its altitude** — founder (COO) → builder-lead (cycle planner) → lifecycle-orchestrator (per-item 0-9 runner) → value-handlers (implement-only), with the "you do not orchestrate, you implement" boundary repeated in every handler.
- **Publish's handler pool is a model of documented dispatch** — a common ILLUSTRATOR SPEC contract, an explicit intent→handler routing table with named tie-breaks ("prefer vector + deterministic over ComfyUI", "motion is a deliberate choice"), and evidence-consulted ComfyUI routing.
- **The BUILD-internal handoff machinery is rigorous** — the context-sentinel protocol defines a typed, validated, accumulating sentinel chain with per-step prerequisite checks, and handoff-schema.md gives a strict YAML payload with validation rules. A cold-start agent can genuinely resume mid-pipeline.
- **lifecycle-orchestrator has robust resumability** — per-phase CHECKPOINT_<phase>.md files, a bounded transient-retry policy with a transient-vs-deterministic classification table, and a repeat-rejection root-cause escalation before the blind 3-strike BLOCK.
- **The IDEA-package contract is thoughtfully designed at the semantic level** — two faces (agent-facing precision + user-facing dossier) projected from one knowledge-parity understanding, with a precise agent-facing exit gate adversarially enforced by an independent challenger agent before handoff.
- **handler-architect is opus-pinned with explicit justification** (wrong pattern = multiplicative downstream cost), a full pattern decision matrix with choose-when/avoid-when, and a verbatim ADR template.
- **The product-lifecycle.md ASSURE/SECURE separation rationale is rigorous** — quality vs security as two distinct first-class gates, mapping each phase to external frameworks (Double Diamond, NPD, PLM, V&V, PDCA/DMAIC), with an unusually precise entry/exit signal table.
- **The self-improvement/inspection machinery actually works** — the publish plugin's own inspector already self-catches the mermaid-specialist naming drift, demonstrating the audit lenses surface real inconsistencies.
- **The glossary's foundry-vs-forge-vs-founder name-collision disambiguation table is exactly the kind of defense the repo needs** and is accurate against the agent/skill files.
- **The value-flow narrative model is internally consistent across README, SLASH_COMMANDS, and the three i2p skills** — the nine-phase cycle with the BUILD⇄ASSURE⇄SECURE loop is explained the same way everywhere a user reads it.

## Key Conclusion

**Can a product travel from "By doing X I propose Y and the value will be Z" to production today? The honest answer is: not autonomously, and not end-to-end.**

The early chain is real and strong. A user can take a raw idea into `/ideate:ideate`, reach knowledge-parity through an adversarially-challenged dialogue, get a build-ready IDEA package, hand it to FOUNDRY, and watch the BUILD engine drive disciplined, test-first, fully-covered implementation through a cleanly-layered orchestration hierarchy. That engine is the marketplace's crown jewel and it genuinely works.

But the journey **stalls at the flagship loop**. The moment a built item reaches ASSURE, the lifecycle freezes: `done ASSURE` — the transition that should advance ASSURE → SECURE — **has no caller anywhere in the marketplace**, even though the catalog assigns it to `/pr-review`. And if a quality or security gate returns NEEDS_REVISION or BLOCK, the `fail` back-edge that should re-enter BUILD is **never invoked by any plugin**. The headline BUILD ⇄ ASSURE ⇄ SECURE loop — advertised identically across README, VALUE_FLOW, and product-lifecycle — is dead wiring: the state machine that tracks it is sound, but nothing drives the two transitions that close it. The statusline `⇄ ×N` cue that would tell an operator "you're on loop pass N, fix and rebuild" never fires because the verb that increments it is never called.

Three secondary breaks compound this for a real user. The **entry point is undiscoverable** — a user with a concrete value-proposition finds the README "Start here →" routes only to browse commands, and the one DELIVER command they're told to type (`/roadmapper`) has no command file. The **DESIGN phase boundary is undefined** — two of the marketplace's own models disagree on whether DESIGN gates BUILD or cross-cuts it, so `done DESIGN` fires on no real boundary in a FLEET pipeline. And the **cycle never closes** — `/operate:iterate` writes an OPPORTUNITY artifact at a path and schema that the DISCOVER consumers don't document ingesting, so the "↻ re-enters DISCOVER" promise has no shared contract.

**The architecture is sound; the wiring is incomplete.** Every defect above is a missing wire or a doc reconciliation, not a structural flaw — there is no broken abstraction to redesign. Fixing the single CRITICAL item (rank 1) makes the advertised loop traversable for the first time; the four HIGH journey/seam items (ranks 2, 7, 8, 9) make the chain end-to-end for a user starting from a raw thesis. Until then, the marketplace is a superb production *house* whose front door, flagship loop, and return path are not yet connected to the engine room. The companion **Improvement Plan** sequences exactly these fixes, CRITICAL-first.
