**Brief**: Opt-in, sandboxed: confirm whether a detected key is still active before classifying it BLOCK,
reducing rotation churn on already-dead secrets. Default OFF; fails safe to BLOCK.

## User stories
- AS a security owner I WANT detected keys checked for whether they are still live before they BLOCK SO
  THAT I'm not forced to rotate already-dead secrets.

## EARS specification
- `EARS-001` (Optional): WHERE live-credential verification is enabled THE SYSTEM SHALL, in a sandbox,
  probe a detected key's validity before finalising its severity.
- `EARS-002` (Event-driven): WHEN a key is confirmed inactive THE SYSTEM SHALL downgrade it from BLOCK
  (still reported as a finding, noted dead) and keep an active key at BLOCK.
- `EARS-003` (Unwanted): IF verification is disabled OR the probe is inconclusive THEN THE SYSTEM SHALL
  treat the key as live (BLOCK) — fail safe, never assume dead.

## Acceptance criteria
1. Given verification enabled and a dead key, When scanned, Then it is reported but not a BLOCK.
2. Given verification disabled or inconclusive, When a key is found, Then it remains a BLOCK.

## Human interface test plan
None — verification is an internal probe behind an opt-in flag. Proof: with a mocked provider probe, a
dead key downgrades (reported, noted dead) and a live/inconclusive/disabled key stays BLOCK (asserted on
the finding's severity). No real network in tests.

## Implementation notes
- **Opt-in only; default OFF** (fail-safe to BLOCK). Sandboxed, rate-limited probes per provider; **never
  exfiltrate the key**. The probe is mocked in tests — no live network (no flakiness, no real secret use).
- Hooks `scan-for-secrets` / the gate verdict synthesis.
