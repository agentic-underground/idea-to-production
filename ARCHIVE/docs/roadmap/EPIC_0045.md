## Shared infrastructure
- **The `comfyui-mcp/` server + container (the buildable core, lands first).** PLAN_0046 builds the
  `server/` (the five MCP tools, allowlist loaders, param validators, `/view` proxy) test-first to the
  coverage floor; PLAN_0049 builds the `container/`. The enforcement (PLAN_0047), boundary hardening
  (PLAN_0048), gate/registration (PLAN_0050), adoption (PLAN_0051) and Phase-2 (PLAN_0052/0053) all refine
  or consume that core — so PLAN_0046 is ordered first and owns it; later PLANs never re-create it.
- **The five MCP tools** (`list_models`, `set_model`, `list_templates`, `submit_prompt`, `get_result`) are
  the shared verb surface; the allowlist semantics PLAN_0047 enforces and the boundary PLAN_0048 hardens
  apply across them; Phase-2 presets (PLAN_0052) are validated parameter sets over the same allowlisted
  templates.
- **Dependency order** (preserved from the legacy children): build (0046) → enforcement (0047) / boundary
  (0048) / container (0049) → gate+docs+registration (0050) → publish adoption (0051); Phase-2 template
  UIs (0052) + pool/cache (0053) follow the Phase-1 core.
- **Rename caveats (load-bearing for the builder).** Legacy item 78 names the **SENTINEL** gate
  (`/security-gate`, `/secret-scan`, `/dependency-audit`) — now the **`security`** plugin
  (`/security:scan-all`, `/security:scan-for-secrets`, `/security:scan-dependencies`). Legacy item 49
  names **PRESSROOM** `handler-comfyui` (`plugins/pressroom/agents/handler-comfyui.md`) — now the
  **`publish`** plugin (`plugins/publish/agents/handler-comfyui.md`). The "flow-server registration
  discipline" item 78 cites is **retired**; follow the current pinned-`.mcp.json` + `PREREQUISITES/40-mcp.md`
  registration discipline that `verify-prereqs` checks C and K enforce.

## Definition of done
- All eight PLANs are `completed` (the engine rolls EPIC completion up from its PLANs).
- `.pipeline/verify` is GREEN for each landed PLAN; `scripts/verify-prereqs.sh` (incl. check P, and — once
  the MCP server is registered in PLAN_0050 — checks C shipped-server parity and K version-pinning) stays
  GREEN.

## Execution contract (for the builder)
The engine builds one PLAN at a time in `## Plans` order, running each PLAN's `## Construction process`
(FOUNDRY scoped to the `(EPIC, PLAN)` pair). The buildable core (PLAN_0046) and container (PLAN_0049) land
before the gate/registration (PLAN_0050) and adoption (PLAN_0051). A failed PLAN leaves its row `engaged`
and resumes next tick; the EPIC lands only when all eight PLANs are GREEN.
