**Brief**: Generate a Software Bill of Materials (CycloneDX and SPDX formats) from the resolved dependency
graph — increasingly a procurement/compliance requirement; reuses the ecosystem detection from
`scan-dependencies`.

## User stories
- AS a vendor responding to a procurement/compliance request I WANT a standards-format SBOM generated from
  my project SO THAT I can supply it without assembling it by hand.

## EARS specification
- `EARS-001` (Ubiquitous): THE SYSTEM SHALL emit a Software Bill of Materials in both CycloneDX and SPDX
  formats from the resolved dependency graph.
- `EARS-002` (Event-driven): WHEN `/security:sbom` runs THE SYSTEM SHALL enumerate every resolved
  dependency (name, version, licence where known) and write a valid CycloneDX and a valid SPDX document.
- `EARS-003` (Unwanted): IF the dependency graph cannot be fully resolved THEN THE SYSTEM SHALL disclose
  the unresolved portion rather than emit an SBOM that silently omits components.

## Acceptance criteria
1. Given a resolved dependency graph, When the SBOM runs, Then a schema-valid CycloneDX and a schema-valid
   SPDX file are produced listing every component.
2. Given a partially-resolvable graph, When run, Then the unresolved components are disclosed.

## Human interface test plan
None — the SBOM is a scan invocation. Proof: a fixture graph yields a schema-valid CycloneDX JSON and a
schema-valid SPDX document listing every component; a partial graph discloses the unresolved portion
(asserted by schema-validating the outputs).
