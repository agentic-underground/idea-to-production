**Brief**: A `spec-reviewer-agent` that checks specs for testability before implementation begins, so
untestable requirements are caught at the spec stage, not after code is written.

## User stories
- AS a builder I WANT specs checked for testability before I implement SO THAT untestable requirements are
  caught at the spec stage, not after code is written.

## EARS specification
- `EARS-001` (Ubiquitous): THE SYSTEM SHALL review a spec for testability before implementation.
- `EARS-002` (Event-driven): WHEN a spec is submitted THE SYSTEM SHALL flag requirements that are
  ambiguous, unmeasurable, or otherwise untestable.

## Acceptance criteria
1. Given a spec with an untestable requirement, When reviewed, Then that requirement is flagged with why.
2. Given a fully-testable spec, When reviewed, Then it passes.

## Human interface test plan
None — the agent is a reviewer over a spec document. Proof: a fixture spec with an ambiguous/unmeasurable
requirement is flagged with the reason; a fully-testable spec passes (asserted on the agent's verdict).

## Implementation notes
- Part of the v2.0 multi-agent pipeline (EPIC_0061); runs pre-implementation over the `code-quality`
  surface. The flag must state *why* a requirement is untestable (ambiguous/unmeasurable), feeding the
  orchestrator (PLAN_0065).
