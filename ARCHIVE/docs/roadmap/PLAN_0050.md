**Brief**: The ship step — the **security** plugin runs the gate on the result (scan-all: PII/secrets/deps/
semgrep) and must PASS before ship; **publish** documents it; the pinned `comfyui` `.mcp.json` entry is
added to the marketplace and listed in `PREREQUISITES/40-mcp.md`. (The `handler-comfyui` curl→MCP switch is
the adoption PLAN_0051.)

## User stories
- AS the marketplace owner I WANT the secured backend gated by security, documented, and registered as a
  pinned MCP server SO THAT it ships to the same standard as every other marketplace surface.

## EARS specification
- `EARS-001` (Ubiquitous): THE SYSTEM SHALL pass the security gate, ship documentation, and register a
  pinned `comfyui` `.mcp.json` entry consistent with `PREREQUISITES/40-mcp.md` and `verify-prereqs` checks
  C and K.
- `EARS-002` (Event-driven): WHEN the security gate returns anything but PASS THE SYSTEM SHALL block the
  ship.

## Acceptance criteria
1. Given the built server, When the security gate runs, Then it passes (PII/secrets/deps/semgrep) before
   ship.
2. Given the marketplace, When inspected, Then a pinned `comfyui` `.mcp.json` entry exists and matches the
   `40-mcp.md` shipped table (check C), with the launcher version pinned (check K).

server; the shipped `.mcp.json` ⟺ `40-mcp.md` parity (check C) and version-pin (check K) both hold after
registration.
