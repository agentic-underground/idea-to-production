**Brief**: Reorganise the marketplace's emergent families around the value-handling paradigm — but
**map first**. Produce a taxonomy map of the repeated patterns (inspector ×9, `/check` ×9, `/self-improve`
×8, handlers ×28, reviewer/critic loops, shared canon), then capture each restructure as its own
independently-shippable sub-item. No moves happen in this slice; the map is the deliverable.

## User stories
- AS the owner I WANT the repeated families named and grouped by characteristic + action SO THAT the
  marketplace structure reflects the value-handling paradigm instead of incidental layout.
- AS a maintainer I WANT a map before any move SO THAT restructuring is deliberate and low-risk.

## EARS specification
- `EARS-001` (Ubiquitous): THE taxonomy map SHALL enumerate every repeated family with its members and a
  proposed grouping.
- `EARS-002` (Event-driven): WHEN the map is approved THE SYSTEM SHALL spawn a sub-item for each
  restructure (e.g. shared `_handlers/`, `_critique/`, `_canon/`), each independently shippable.
- `EARS-003` (Unwanted): IF a proposed move would break a plugin's self-contained installability THEN
  THE SYSTEM SHALL flag it and not proceed without a path that preserves `${CLAUDE_PLUGIN_ROOT}`-only
  resolution.

## Acceptance criteria
1. A taxonomy map document exists listing families (inspectors, checks, self-improve, handlers,
   reviewers, canon) with their members + a proposed grouping.
2. Each restructure is captured as its own sub-item with acceptance criteria.
3. No proposed restructure violates the self-contained-plugin or canonical-copy promises.

## Human interface test plan
None — this slice produces a documentation artefact (the taxonomy map) and sub-item specs; it adds no
user-facing UI element. The "proof" is the map's completeness and conformance, checked against the survey.

## Implementation notes
- **Survey** `plugins/*/{agents,skills,commands}` to enumerate the families and count members.
- **Respect the invariants** — the canonical-copy promise (KAIZEN, `check.sh`, `inject-kaizen.sh`
  mirrored byte-identical, CI-verified by `scripts/verify-prereqs.sh`) and the self-contained-plugin rule
  (`${CLAUDE_PLUGIN_ROOT}`-only path resolution). The map flags any move that would break either.
- **Reference** the value-station / value-handler definitions in
  [`plugins/foundry/VALUE_FLOW.md`](../../plugins/foundry/VALUE_FLOW.md).
- **Map only** — this slice does not move files; restructures are emitted as sub-items (future PLANs)
  for deliberate, low-risk execution.
