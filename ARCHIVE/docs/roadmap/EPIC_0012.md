## Shared infrastructure
- **The MCP config surface** — `~/.claude/.mcp.json` (where Claude Code loads MCP servers at startup) and
  the hook-state convention under `~/.claude/hook-state/`. PLAN_0013 establishes the read/merge discipline
  over `.mcp.json` and the hook-state flag pattern; PLAN_0014's exposure policy reasons about the same
  verb surface those servers expose. Neither PLAN writes into a user repo — both operate on `~/.claude`
  state only.
- **No cross-PLAN code dependency** — the two slices are independently shippable (an installer prompt vs.
  a verb-exposure policy). They are grouped because both harden how the marketplace presents its MCP
  surface; the engine may build them in either order.

## Definition of done
- Both PLANs are `completed` (the engine rolls EPIC completion up from its PLANs).
- `.pipeline/verify` is GREEN for each landed PLAN; `scripts/verify-prereqs.sh` check P stays GREEN.

## Execution contract (for the builder)
The engine builds one PLAN at a time in `## Plans` order, running each PLAN's `## Construction process`
(FOUNDRY scoped to the `(EPIC, PLAN)` pair). A failed PLAN leaves its row `engaged` and resumes next tick;
the EPIC lands only when both PLANs are GREEN.
